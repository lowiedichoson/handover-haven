# Reverse Bridge Entries

## Who Asks

| | |
|---|---|
| **Requested by** | Accounting or TCSG Department |
| **How they ask** | Email, Sapphire Ticket, Walk-up |
| **Frequency** | Ad-hoc |

## What It Is

Reversing bridge entries is a crucial step done from the backend because as of writing, ***we don't have a capability to do it in any of our systems that has a user interface***. This is done because both the TCSG & Accounting department has verified that there's a discrepancy in the entries generated by the Esettlement system.

## When This Request Happens
- The **current** entries generated are **incorrect** as verified by TCSG & Accounting department.
- The incorrect entries are already ***transferred and posted in Navision***, therefore requiring us to manually insert a reversed *(reversed signs)* version of them.

## Prerequisites

- Access to the Navision server and `[EBIZ PROD DATABASE]` database in SQL Server Management Studio (SSMS)
- Access to execute queries with `INSERT` keyword
- Completed Job Request Form (JRF) and Release Notes
- Approved JRF (usually Infra Team Lead and/or the concerned Department's Team Lead)

## Steps

> ⚠️ **Before you start**, determine the state of the existing entries:
>
> - **Not yet transferred to Navision?** You don't have to reverse anything, because it's not in Navision in the first place. Ask TCSG to reprocess their transactions in Esettlement to get the transactions with correct amounts.
> - **Already in Navision?** Reverse the entries in Navision first so it's as if they never existed *(this is why this documentation exists)*.
> - **Do this in a staging environment PRIOR** to proceeding in production environment.
> Once the above is resolved, proceed with the steps below.

### 1. Verification of the incorrect entries

Check if the incorrect entries are existing in the `[dbo].[E-Business Services, Inc_$JournalVoucherHistory]` table of `[EBIZ PROD DATABASE]` database. ***This is a history table where posted entries gets transferred into.***

```sql
DECLARE @TransactionDate DATETIME = '2025-01-02'; -- update if necessary
DECLARE @DocumentType VARCHAR(20) = 'Bridge'; -- filter bridge entries only

SELECT 
	--[timestamp],
	[DocumentType],
	--[EntryNo],
	[TransactionDate],
	[DocumentNo],
	[AccountType],
	[AccountNo],
	[Description],
	[Amount],
	[DimensionCode],
	[ProductCode],
	[CurrencyCode],
	[ExchangeRate],
	[Posted],
	[brsaid],
	[BranchSA],
	[DeductedAmount],
	[Rate Type],
	[JournalType],
	[VATIdentifier],
	[JournalBatchName],
	[GenPostingType],
	[VATBusPostingGroup],
	[VATProdPostingGroup],
	[WHTBusPostingGroup],
	[WHTProdPostingGroup],
	[SourceDocumentType]
FROM [EBIZ PROD DATABASE].[dbo].[E-Business Services, Inc_$JournalVoucherHistory]
WHERE [TransactionDate] = @TransactionDate
AND [DocumentType] = @DocumentType
ORDER BY [DocumentNo] ASC;

```

### 2. Reverse the incorrect entries 

Reversing the incorrect entries require you to insert the **same incorrect entries** but this time, their signs are ***reversed***. 

```sql
-- Author: Lowie
-- Created date: 2025-05-07
-- Description:
-- [E-Business Services, Inc_$JournalVoucher] - table where the entries gets saved before posting
-- [E-Business Services, Inc_$JournalVoucherHistory] - table where the posted entries get saved
-- reversing happens by reinserting the same entries but with reversed signs (positive becomes negative, negative becomes positive)
DECLARE @TransactionDate DATETIME = '2025-01-02'
DECLARE @DocumentType VARCHAR(20) = 'Bridge'

INSERT INTO [EBIZ PROD DATABASE].[dbo].[E-Business Services, Inc_$JournalVoucher] (
	--[timestamp], 
	[DocumentType],
	--[EntryNo],
	[TransactionDate],
	[DocumentNo],
	[AccountType],
	[AccountNo],
	[Description],
	[Amount],
	[DimensionCode],
	[ProductCode],
	[CurrencyCode],
	[ExchangeRate],
	[Posted],
	[brsaid],
	[BranchSA],
	[DeductedAmount],
	[UserID],
	[Rate Type],
	[JournalType],
	[VATIdentifier],
	[JournalBatchName],
	[GenPostingType],
	[VATBusPostingGroup],
	[VATProdPostingGroup],
	[WHTBusPostingGroup],
	[WHTProdPostingGroup],
	[SourceDocumentType],
	[GlobalDimensionCode],
	[Country_ID])
SELECT 
	--[timestamp],
	[DocumentType],
	--[EntryNo],
	[TransactionDate],
	'R' + LTRIM(RTRIM([DocumentNo])) [DocumentNo], -- R for identification that this is a reversal entry
	[AccountType],
	[AccountNo],
	[Description],
	([Amount] * -1) [Amount],
	[DimensionCode],
	[ProductCode],
	[CurrencyCode],
	[ExchangeRate],
	[Posted],
	[brsaid],
	[BranchSA],
	[DeductedAmount],
	'System' [UserID], -- System for identifier that this was a data patch
	[Rate Type],
	[JournalType],
	[VATIdentifier],
	[JournalBatchName],
	[GenPostingType],
	[VATBusPostingGroup],
	[VATProdPostingGroup],
	[WHTBusPostingGroup],
	[WHTProdPostingGroup],
	[SourceDocumentType],
	'' [GlobalDimensionCode], -- empty, this column does not exist in the history table
	'' [Country_ID] -- empty, this column does not exist in the history table 
FROM [EBIZ PROD DATABASE].[dbo].[E-Business Services, Inc_$JournalVoucherHistory]
WHERE [TransactionDate] = @TransactionDate
AND [DocumentType] = @DocumentType
ORDER BY [DocumentNo] ASC

```

### 3. Execute the reversal script

The server / database administrator will execute the reversal script with your assistance.

### 4. Verify that the reversal script ran successfully

There are several ways of verifying that the script was successful, they are as follows:

a. The query returned the rows affected and no errors were encountered during its execution.

b. Run the following script to fetch the newly inserted data in `[dbo].[E-Business Services, Inc_$JournalVoucher]` table.

```sql
DECLARE @TransactionDate DATETIME = '2025-01-02'
DECLARE @DocumentType VARCHAR(20) = 'Bridge'

SELECT 
	*
FROM [dbo].[E-Business Services, Inc_$JournalVoucher]
WHERE [TransactionDate] = @TransactionDate
AND [DocumentType] = @DocumentType
AND [DocumentNO] LIKE 'R%'
ORDER BY [Description] ASC;
```

### 5. Next steps

After the incorrect bridge entries have been successfully reversed, the most likely next step is to [Re/create Entries.](001-recreation-of-bridge-entries.md)
